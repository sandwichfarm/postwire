## Objective

Enable multi-stream-per-channel operation under an opt-in flag while preserving the single-stream default unchanged.

## Tasks

### Task 1: Refactor Channel to hold multiple sessions

<read_first>
- .planning/phases/08-multiplexing/08-CONTEXT.md
- src/channel/channel.ts (specifically the #session field, openStream, onStream, frame routing, close, stats)
- src/session/index.ts
- src/framing/types.ts
- src/types.ts
</read_first>

<action>
Refactor `Channel` so it hosts a map of sessions keyed by stream ID. Key changes:

1. Replace `#session: Session | null` with `#sessions: Map<number, Session>`.
2. Add `#nextStreamId: number` initialized based on role (initiator=1, responder=2, increment by 2 on openStream).
3. Add `#options.multiplex?: boolean` to `ChannelOptions` (default false).
4. In CAPABILITY handshake, include `multiplex` in the negotiated capabilities; merged = local && remote. Only activate multiplex if merged is true.
5. Update `channel.openStream()`:
   - If `!mergedMultiplex`: preserve current single-stream behavior (streamId=0, only one session allowed; throw if called twice).
   - If `mergedMultiplex`: allocate next streamId, create a Session with that ID, store in `#sessions`, return stream handle.
6. Update `onmessage` → inbound OPEN:
   - Single-stream mode: create session with streamId=0 exactly as before.
   - Multiplex mode: create session on demand, keyed by the frame's streamId.
7. Update frame routing for DATA/CREDIT/CANCEL/RESET/CLOSE: dispatch to `#sessions.get(frame.streamId)`.
8. Update `channel.close()`: close all sessions in map, clear map.
9. Update `channel.stats()`: return per-stream stats for each session in the map; single-stream shape unchanged when only one session exists.
10. The raw-frame hooks (`onRawDataFrame`, `onRawControlFrame`, `sendRawFrame`) continue to work — they don't care about session multiplexing.

All existing tests must pass unchanged (multiplex defaults to false). Add test in `tests/unit/channel/multiplex.test.ts`:
- `multiplex: true` both sides → CAPABILITY negotiates multiplex:true
- `multiplex: true` one side + `multiplex: false` other → merged false, falls back to single-stream
- Open 3 streams when multiplex enabled; each gets a unique streamId
- Open 2 streams when single-stream mode throws on the second
</action>

<action>
Create `tests/integration/multiplex-hol.test.ts`:

1. Create two Channels over MessageChannel with `{ multiplex: true }` on both.
2. Await CAPABILITY handshake.
3. Open 4 concurrent streams on the initiator side. Start writing 32 chunks (1 KB each) on each stream.
4. On the responder side, set up handlers:
   - Stream 1, 3, 4: consume chunks normally, count them
   - Stream 2: NEVER consume (leave the session's reorder buffer to fill up and credits to drain)
5. Start a 2-second timer.
6. After 2 seconds (use `setTimeout`), assert:
   - Stream 1 received all 32 chunks
   - Stream 3 received all 32 chunks
   - Stream 4 received all 32 chunks
   - Stream 2 received < 32 chunks (it's intentionally stalled)
   - `channel.stats()` on the initiator side shows all 4 streams with different creditWindowAvailable values
7. Close all 4 streams and the channels; verify clean teardown.

Commit `feat(08-01): Channel multiplex via per-stream session map + HoL test`.
</action>
